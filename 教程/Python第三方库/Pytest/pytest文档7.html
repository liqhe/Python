用属性标记测试函数¶

通过使用pytest.mark帮助程序，您可以轻松地在测试功能上设置元数据。有一些内置标记，例如：

  • skip - 总是跳过测试功能
  • skipif - 如果满足某个条件，则跳过测试函数
  • xfail - 如果满足某个条件，则产生“预期失败”结果
  • 参数化以执行对同一测试函数的多次调用。

可以轻松创建自定义标记或将标记应用于整个测试类或模块。有关也用作文档的示例，请参阅使用自定义标记。

注意

标记只能应用于测试，对夹具没有影响 。

未知标记提高错误：-strict ¶

当--strict命令行标志传递，与所施加的任何未知标记@pytest.mark.name_of_the_mark装饰器将触发一个错误。由pytest或插件定义或添加的标记不会触发错误。

标记可以pytest.ini像这样注册：

[pytest]
markers =
    slow
    serial

这可以用于防止用户意外错误标记标记名称。想要强制执行此操作的测试套件应添加--strict到addopts：

[pytest]
addopts = --strict
markers =
    slow
    serial

标记修改和迭代¶

版本3.6中的新功能。

pytest的标记实现传统上通过简单地更新__dict__函数的属性来累积添加标记。结果，标记将以令人惊讶的方式无意中沿着类层次结构传递。此外，用于检索它们的API是不一致的，因为来自参数化的标记将与使用@pytest.mark装饰器和通过添加的标记应用的标记不同地存储node.add_marker。

这种状态使技术上几乎不可能正确使用来自标记的数据，而无需深入了解内部结构，从而导致在更高级的用法中出现细微且难以理解的错误。

根据标记声明/更改的方式，可以获得MarkerInfo可能包含兄弟类标记的标记， MarkDecorators标记来自参数化或来自node.add_marker调用，丢弃先前标记。MarkerInfo当它实际上表示具有相同名称的多个标记上的合并视图时，也像单个标记一样。

最重要的是，模块，类和函数/方法无法以相同的方式访问标记。实际上，标记只能在函数中访问，即使它们是在类/模块上声明的。

在pytest 3.6中引入了一个访问标记的新API，以解决初始设计中的问题，提供_pytest.nodes.Node.iter_markers()以一致的方式迭代标记并重新处理内部的方法，这解决了初始设计的大量问题。

更新代码¶

旧Node.get_marker(name)函数被视为已弃用，因为它返回一个内部MarkerInfo对象，该对象包含合并名称*args以及**kwargs适用于该节点的所有标记。

一般来说，有两种方案可以处理标记：

1.标记互相覆盖。订单很重要，但您只想将您的商标视为单个商品。例如 log_level('info')，在模块级别可以覆盖log_level('debug')特定测试。

在这种情况下，使用Node.get_closest_marker(name)：

# replace this:
marker = item.get_marker("log_level")
if marker:
    level = marker.args[0]

# by this:
marker = item.get_closest_marker("log_level")
if marker:
    level = marker.args[0]

2.标记以加成方式组成。例如，skipif(condition)标记意味着您只想评估所有这些，订单甚至不重要。您可能希望将您的标记视为一组。

在这种情况下，迭代每个标记*args并**kwargs单独处理它们。

# replace this
skipif = item.get_marker("skipif")
if skipif:
    for condition in skipif.args:
        # eval condition
        ...

# by this:
for skipif in item.iter_markers("skipif"):
    condition = skipif.args[0]
    # eval condition

如果您不确定或有任何疑问，请考虑 解决问题。

相关问题¶

以下是新实施修复的问题的非详尽列表：

  • 标记不会选择嵌套类（＃199）。
  • 标记在所有相关类别上染色（＃568）。
  • 组合标记 - 计算args和kwargs（＃2897）。
  • request.node.get_marker('name')返回None类中应用的标记（＃902）。
  • 在参数化中应用的标记存储为markdecorator（＃2400）。
  • 以向后不兼容的方式修复标记交互（＃1670）。
  • 重构标记以摆脱当前的“标记转移”机制（＃2363）。
  • 介绍FunctionDefinition节点，在generate_tests中使用它（＃2522）。
  • 删除命名标记属性并收集项目中的标记（＃891）。
  • 来自参数化的跳过标记隐藏模块级跳过标记（＃1540）。
  • skipif + parametrize不跳过测试（＃1296）。
  • 标记转移与继承不兼容（＃535）。

更多细节可以在原始PR中找到。
